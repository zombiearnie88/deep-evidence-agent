# Incremental Compile For New Documents Implementation Checklist

## Goal

Change compile so that the default behavior only processes newly added or
otherwise uncompiled documents, instead of recompiling every indexed document in
the workspace on each run.

This change should eliminate unnecessary token spend while preserving the parts
of compile that legitimately update shared outputs, such as:

- backlink patches on older summary pages
- shared evidence pages rendered from the full manifest corpus
- cross-document conflict detection against existing regulation/procedure pages

## Root Cause

The current compile flow treats `list_documents(workspace)` as both:

- the set of documents to compile now
- the effective active corpus for every summarization/planning step

That means a second compile after ingesting document `B` will still:

1. summarize document `A` again
2. rewrite `wiki/summaries/<A>.md` from scratch
3. then patch backlinks

Backlink updates on older summaries are acceptable. The bug is that the summary
body for older documents is regenerated even when those documents were not the
reason for the compile.

## Locked Decisions

- Default compile behavior is `new only`.
- A compile run targets documents whose `DocumentRecord.status != "compiled"`.
- If there are no target documents, compile returns success as a no-op with
  `processed_files = 0`.
- Older compiled documents remain available as read-only compile context through
  existing summaries, wiki pages, and evidence manifests.
- Older summary pages may still be patched in compiler-managed sections such as
  `Derived Pages`.
- Older summary bodies must not be regenerated by the summarization stage.
- Do not add a separate full-workspace rebuild mode in this phase.
- Do not force shared API schema changes unless a concrete external response
  shape change becomes necessary.

## Non-Goals

- Do not redesign the taxonomy or evidence architecture.
- Do not reintroduce workspace-wide summary regeneration.
- Do not block backlink updates to older summaries when shared pages or evidence
  legitimately reference them.
- Do not change the meaning of `compiled` beyond compile-target selection in this
  phase.

## Target Behavior

Example sequence:

1. Ingest document `A`.
2. Compile workspace.
3. `A` is summarized, planned, drafted, verified, and marked `compiled`.
4. Ingest document `B`.
5. Compile workspace again.
6. Only `B` goes through summarization and downstream LLM stages.
7. Existing summaries/pages/manifests from `A` are loaded as context from disk.
8. Shared outputs may still be updated if `B` changes them.
9. `wiki/summaries/<A>.md` may receive managed backlink updates, but its summary
   body is not regenerated.
10. A third compile with no new documents returns a successful no-op.

## Design Overview

### Two Distinct Sets

The implementation should separate:

- `target_documents`: documents compiled in the current run
- `existing_context`: previously compiled workspace state loaded from disk

Only `target_documents` should enter:

- materialization
- summary generation
- evidence planning/drafting/verification for the new document
- taxonomy planning/drafting for the new document

`existing_context` should be read from:

- `wiki/summaries/`
- `wiki/topics/`, `wiki/regulations/`, `wiki/procedures/`, `wiki/conflicts/`
- `.brain/evidence/by-document/`
- rendered `wiki/evidence/`

### Backlink Rule

Backlink application should no longer assume that only summaries written in the
current run can change.

Instead:

- `summary_to_links` may contain both new and older summary slugs
- backlinking should patch every summary slug present in `summary_to_links`
- summary patching must operate via managed section replacement only
- summary body generation must remain scoped to `target_documents`

### No-Op Rule

If there are no `target_documents`, compile should:

- create or update the compile job as usual
- return success
- report `processed_files = 0`
- avoid LLM calls
- avoid changing summary bodies

## Implementation Phases

## Phase 1: Compile Target Selection

Files:
`services/evidence-compiler/src/evidence_compiler/api.py`

Checklist:
- [ ] Add a helper or local selection block that separates `all_documents` from
  `target_documents`.
- [ ] Use `status != "compiled"` as the compile target rule.
- [ ] Keep `all_documents` available for status/logging if needed, but only pass
  `target_documents` into compile execution.
- [ ] If `target_documents` is empty, return a successful no-op result with
  `processed_files = 0`.
- [ ] Update job payload fields so they describe the actual compile target set,
  not the whole workspace.
- [ ] Mark only `target_documents` as `compiled` after a successful run.

## Phase 2: Incremental Compiler Pipeline Scope

Files:
`services/evidence-compiler/src/evidence_compiler/compiler/pipeline.py`

Checklist:
- [ ] Keep `compile_documents(...)` scoped to the provided `target_documents`
  only.
- [ ] Ensure materialization and summarization loops run only for
  `target_documents`.
- [ ] Ensure evidence planning/drafting/verification loops run only for
  `target_documents`.
- [ ] Ensure taxonomy planning/drafting loops run only for `target_documents`.
- [ ] Continue reading existing wiki briefs, evidence manifests, and rendered
  evidence pages from disk as read-only context for those new targets.

## Phase 3: Summary Write Isolation

Files:
`services/evidence-compiler/src/evidence_compiler/compiler/pipeline.py`

Checklist:
- [ ] Keep `_write_summary_page(...)` calls limited to `target_documents`.
- [ ] Confirm that older summary pages are never passed back through summary
  generation when they are not compile targets.
- [ ] Preserve the ability to patch managed sections like `Derived Pages` on
  older summaries.
- [ ] Ensure summary body regeneration and backlink patching are clearly
  separated in code.

## Phase 4: Backlinking Older Summaries Safely

Files:
`services/evidence-compiler/src/evidence_compiler/compiler/pipeline.py`

Checklist:
- [ ] Change summary backlink application to iterate over summary slugs present
  in `summary_to_links`, not just `artifacts.summaries`.
- [ ] Resolve each summary slug to its on-disk summary page path.
- [ ] Patch only the managed `Derived Pages` section on those summaries.
- [ ] Do not regenerate summary body content for older summaries during backlink
  updates.
- [ ] Keep backlink patching idempotent.

## Phase 5: Watch / Queue Behavior

Files:
`services/brain-service/src/brain_service/watch_manager.py`
`services/brain-service/src/brain_service/main.py`

Checklist:
- [ ] Queue compile jobs using only uncompiled/new document hashes.
- [ ] Update queued job payload counts to reflect only target documents.
- [ ] Keep follow-up watch-triggered compile jobs incremental as well.
- [ ] Preserve current coalescing behavior when a compile is already running.
- [ ] Keep the external `/jobs/compile` endpoint behavior unchanged aside from
  incremental scope.

## Phase 6: Optional Write-Churn Reduction

Files:
`services/evidence-compiler/src/evidence_compiler/compiler/pipeline.py`

Checklist:
- [ ] Consider adding a write-if-changed guard for `_write_page(...)` or a local
  wrapper.
- [ ] Ensure this optimization does not change semantics for managed-section
  updates.
- [ ] Treat this as a reviewability improvement, not the primary fix.

## Phase 7: Tests

Files:
`tests/integration/test_milestone_a.py`

Checklist:
- [ ] Add an incremental compile scenario: compile `A`, ingest `B`, compile
  again, assert only `B` is processed.
- [ ] Assert `wiki/summaries/<A>.md` does not have its summary body regenerated
  during the second compile.
- [ ] Allow and assert managed backlink changes on `A` when shared outputs now
  reference it.
- [ ] Add a no-new-documents compile scenario that succeeds with
  `processed_files = 0`.
- [ ] Update rerun expectations that currently assume every compile reprocesses
  the same document again.
- [ ] Add watch-mode coverage showing follow-up compile targets only newly added
  documents.

## Phase 8: Quality Gate

Checklist:
- [ ] Run `pytest tests/integration/test_milestone_a.py`.
- [ ] Run any watch-related integration tests affected by the compile target
  change.
- [ ] Run `npx basedpyright`.
- [ ] If any shared API shape changes are introduced unexpectedly, update the
  corresponding frontend types and run `pnpm typecheck`.

## Definition Of Done

- [ ] Compile defaults to processing only documents whose status is not
  `compiled`.
- [ ] Compiling new document `B` does not regenerate the summary body for older
  compiled document `A`.
- [ ] Older summaries may still receive managed backlink updates when required.
- [ ] A compile with no new documents succeeds as a no-op with
  `processed_files = 0`.
- [ ] No LLM summary/evidence/taxonomy stages run for already compiled
  documents.
- [ ] Existing compiled documents remain usable as read-only context through
  workspace state on disk.
- [ ] Watch-triggered compile jobs remain incremental.
- [ ] Tests and typecheck pass.
